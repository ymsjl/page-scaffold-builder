边框和控制点的作用是调整组件的大小和位置，但是不是所有的组件都可以调整大小和位置，这取决于组件的属性设置。有一些组件可能只能调整位置，而不能调整大小。为了让布局更灵活，

功能组件最好是流体宽度的，这样它们可以适应不同的屏幕尺寸和容器大小。比如表格组件，列表组件，卡片组件等，这些组件通常会占满父容器的宽度，以提供更好的用户体验。

还要有容器组件，容器组件是用来包裹其他组件的，它们可以提供布局和样式的功能。容器组件可以是流体宽度的，也可以是固定宽度的，这取决于设计需求。容器组件可以包含其他组件，也可以包含部件。

部件一定是组件中的视觉元素，可以拥有自己的框，可以被编辑，移除，复制，添加以及被拖动。比如表格组件中的列，实际上他们是通过组件的 `columns` 属性来定义的。`columns` 代表了表格中的所有列以及表格的查询表单，为了能更直观地编辑表格组件中的列，我们希望用户就可以直接在界面上通过操作 `columns[number]` 所渲染出的列来编辑 `columns` 属性，反过来 `columns[number]` 的变化也会直接反映在界面上。这样用户就可以通过操作部件来编辑组件的属性，从而实现更直观的编辑体验。

`ProForm` 表格组件也是同理，他们都采用数据声明式的方式来定义组件的结构和内容，但是我们知道这些数据都有对应的界面元素来展示，所以我们希望用户可以直接通过操作界面元素来编辑这些数据，从而实现更直观的编辑体验。

与之不同的是 `ProTable` 中的 `toolBar` 属性，虽然同为数据声明式的方式来定义组件的结构和内容，但是 `toolBar` 中的元素的类型是 React 节点，而不是抽象的 Schema 定义，所以它的地位相当于组件内部的子组件，他应该在组件节点树中占有一个位置，可以通过组件节点数来定位到这个子组件的位置，从而实现对这个子组件的编辑和操作。因为一个编写好的组件 `props` 中哪些 `ReactNode` 类型的属性在哪个地方渲染是定义好的，所以我们可以将这些 `ReactNode` 替换成经由我们包装过的组件，为这个地方提供组件的添加/编辑/删除的功能，这样用户就可以通过操作界面元素来编辑这些 `ReactNode`，从而实现更直观的编辑体验。相当于是一些开槽，我们要做的是将这些槽位暴露出来，让用户可以在这些槽位上添加组件，并且在每个槽位上，也就是被添加的组件外包装一个组件，这个组件的作用是提供编辑/删除/复制/拖动等功能，同时这个包装组件不能影响槽内组件的渲染。这样用户就可以通过操作界面元素来编辑这些 `ReactNode`，从而实现更直观的编辑体验。这对 UI/UX 设计是非常充满挑战的，当前的画布方案是直接渲染原组件，而不是提供一个纯视觉的占位组件，这样的好处是用户可以直接在界面上看到组件的真实样子，从而更直观地编辑组件的属性和内容。这样加重了组件编辑功能的复杂度，因为我们需要在原组件的基础上添加编辑功能，同时还要保证组件的正常渲染和交互，这对 UI/UX 设计是非常充满挑战的。设想如果采用 canvas 方案，用户在界面上看到的只是一个占位组件，相当于每一个组件都会有一个对应的视觉替代版本，这些视觉替代盒子不太可能还原原组件的样式和交互，但是可以更轻易地添加编辑UI，我可以提供一个预览功能，用户在编辑组件时看到的是占位组件，但是可以通过预览功能来查看组件的真实样子，这样就可以兼顾编辑功能的复杂度和用户体验了。
